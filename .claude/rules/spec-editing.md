When editing any spec file, follow these two constraints:

**Constraint 1: Discussion entry for every substantive change.**

When a change alters a decision, adds or removes a capability, or changes behavior described in the spec, add a new entry in that file's `## Discussion` section. Use the existing format:

```
**Q: Why was {description of what changed}?**

{Short answer + explanation.}
```

This applies regardless of whether the spec file was generated by the current skill or a previous one. Formatting fixes, typo corrections, and rewording that does not change meaning do not need a Discussion entry.

**Constraint 2: Specs reflect current state only.**

The spec body must only describe what the component does now. It must never contain phrasing like "previously X, now Y" or "X and NOT Y (which was the old behavior)." Comparisons to past decisions, removed alternatives, and migration notes belong exclusively in the `## Discussion` section. When updating a spec section, replace the old content with the new content cleanly.
